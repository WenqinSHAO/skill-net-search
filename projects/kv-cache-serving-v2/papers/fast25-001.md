# fast25-001 — Mooncake

## Why it matters in this project

Mooncake is one of the strongest architectural anchors in the project because it is not just a point mechanism. It presents a KV-cache-centric serving platform and makes KV the organizing principle of the whole system.

## Architectural role

**Role**: multi-tier, KV-centric production serving architecture.

Mooncake goes beyond phase disaggregation. It says that once KV dominates the serving bottleneck, the architecture should be organized around:

- where KV resides
- how KV moves across tiers
- how scheduling is driven by KV state
- how overload is handled when KV becomes the limiting resource

This is a fuller systems architecture than papers that only optimize one phase or one memory tactic.

## What architectural problem it solves

Mooncake addresses the production version of the problem:

- long-context serving under overload
- finite GPU memory
- underused CPU/DRAM/SSD resources
- the need to satisfy latency SLOs without assuming all work can be admitted

Its architecture treats KV as the scarce state that must be jointly managed across compute, memory tiers, and scheduler policy.

## Placement in the design space

This paper belongs to:

- disaggregated serving
- tiered KV memory
- KV-aware scheduling
- production-oriented platform design

Among the papers in this project, Mooncake is one of the clearest examples of KV becoming the center of the overall serving architecture rather than one subsystem.

## Architectural consequence

Mooncake expands the architectural surface of the field in three ways:

1. KV is no longer a per-request local cache; it is a distributed, tiered service resource.
2. Scheduler decisions are explicitly conditioned on KV behavior.
3. Admission control and overload handling become part of KV architecture, not just queue management.

This is a major step beyond papers that focus only on compression or only on phase separation.

## How to compare it to later papers

Use Mooncake as the platform-level reference when comparing:

- DistServe: phase disaggregation without the same depth of KV-centric storage architecture
- HCache: state restoration and caching angles within serving infrastructure
- Prefill-as-a-Service / DualMap: later architectural refinements around service decomposition and placement

## Project takeaway

Mooncake is one of the papers that most strongly supports the project’s central thesis:

**the field evolves toward KV-aware serving only when KV becomes the governing resource in architecture, scheduling, and storage together.**
