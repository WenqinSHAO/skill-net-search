# nsdi26-008 — DroidSpeak

## Why it matters in this project

DroidSpeak is architecturally important because it extends KV reuse beyond the usual same-model, same-prefix setting. It moves the project from “reuse across requests” to “reuse across collaborating models.”

## Architectural role

**Role**: cross-model KV sharing architecture.

The key architectural move is to reinterpret KV cache as a transferable intermediate state between distinct LLM executions, not just between repeated invocations of one model.

That matters because it opens a new branch in the design space:

- KV as local cache
- KV as disaggregated state
- KV as reusable cross-model interface

DroidSpeak mainly occupies the third branch.

## What architectural problem it solves

In compound and multi-LLM systems, multiple models often consume overlapping context. Without sharing, each model repeats prefill work independently.

DroidSpeak addresses the structural inefficiency:

- redundant prefill across model boundaries
- repeated reconstruction of nearly compatible KV state
- lack of a mechanism for quality-preserving cross-model reuse

Its answer is not exact reuse. It is selective reuse plus selective recomputation.

## Placement in the design space

This paper belongs to:

- KV sharing and reuse
- distributed serving
- multi-model / agentic serving architecture

It is not primarily a storage-tier or retrieval paper. Its importance is at the interface between multiple executions that share architectural compatibility but not full identity.

## Architectural consequence

DroidSpeak broadens what the system may consider a reusable state boundary.

The project-level significance is that it turns KV cache into a possible **inter-model protocol object**:

- part reused
- part recomputed
- overall benefit realized at system level

This is a qualitatively different move from same-request memory optimization.

## How to compare it to other papers

Useful comparison axes:

- with LMCache / KVShare / SemShareKV:
  - all are about reuse, but DroidSpeak is stronger on cross-model architectural reuse
- with Q-KVComm:
  - DroidSpeak treats KV as reusable serving state
  - Q-KVComm treats KV more like a communication/compression object for agents
- with Mooncake or DistServe:
  - those papers restructure serving around KV placement
  - DroidSpeak restructures reuse boundaries around KV compatibility

## Project takeaway

DroidSpeak matters because it suggests that future KV-serving architectures may not stop at cache efficiency. They may use KV state as a **shared intermediate representation across cooperating models**.
