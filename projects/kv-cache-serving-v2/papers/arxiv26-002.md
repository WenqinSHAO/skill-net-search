# arxiv26-002 — DualMap

## Why it matters in this project

DualMap matters because it addresses a scheduling conflict that appears only after KV reuse becomes a cluster-level optimization target: cache affinity and load balancing pull in opposite directions.

## Architectural role

**Role**: KV-aware routing and placement policy for distributed serving.

The paper does not primarily change the memory layout or compression method. It changes the request-to-instance mapping policy under distributed serving.

Architecturally, this is important because it treats KV reuse as a placement concern rather than only a local cache concern.

## What architectural problem it solves

The core conflict is:

- cache-affinity wants similar-prefix requests co-located
- load-balancing wants requests spread evenly

Most systems can only privilege one side at a time. DualMap tries to retain both by introducing a dual candidate mapping and then choosing adaptively based on system state.

## Benchmarking structure

This paper is benchmarked as a **distributed scheduling** paper.

### Scenario

- distributed LLM serving with shared-prefix reuse opportunities
- dynamic, skewed, real-world workloads
- elastic scaling and hotspot conditions

### Workload shape

- repeated prefixes with reuse value
- hotspot and skew conditions
- cluster instances with changing load

### Metrics

- effective request capacity
- TTFT under SLO constraints
- robustness under hotspots
- scaling/remapping overhead

### Why this matters for later comparison

DualMap should mainly be compared to:

- cache-affinity schedulers
- load-aware schedulers
- future KV-aware global schedulers

It is closer to Mooncake on scheduler centrality than to IceCache or TRIM-KV on memory management mechanics.

## Placement in the design space

This paper belongs to:

- distributed serving
- cache-aware routing
- KV reuse as a placement primitive

## Project takeaway

DualMap teaches that once reuse matters at cluster scale, the architecture must explicitly answer:

**what should the scheduler optimize first: locality, balance, or SLOs?**

The paper’s contribution is to make that tradeoff an explicit routing design problem.
