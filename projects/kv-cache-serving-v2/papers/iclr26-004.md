# iclr26-004 — LouisKV

## Why it matters in this project

LouisKV is a strong representative of the retrieval branch because it is explicit about long input-output scenarios and about reducing the orchestration costs of retrieval itself.

## Architectural role

**Role**: semantic-boundary-triggered KV retrieval architecture.

Compared with earlier retrieval papers, LouisKV is more explicit about:

- when retrieval should happen
- how retrieval units should differ for input and output sequences
- how systems overhead can dominate if retrieval is too frequent or too coarse

Architecturally, it is not just a token-importance method. It is a retrieval-trigger and retrieval-granularity design.

## What architectural problem it solves

The problem is that retrieval-based KV serving often pays too much overhead:

- per-token retrieval is too expensive
- coarse page-level retrieval is too inaccurate
- long-output reasoning workloads make these costs worse

LouisKV tries to make retrieval happen at semantically meaningful boundaries and under different structures for prompt versus generated output.

## Benchmarking structure

This paper is benchmarked as a **long input-output retrieval** paper.

### Scenario

- long-sequence inference with retrieval-based KV management
- especially scenarios with long outputs, not only long prompts

### Workload shape

- long-input short-output
- short-input long-output
- long-input long-output

This breakdown is especially useful for the project because many papers silently optimize only one of these shapes.

### Metrics

- speedup over prior retrieval methods
- near-lossless accuracy retention
- effectiveness across multiple sequence-shape regimes

### Why this matters for later comparison

LouisKV should be compared to:

- InfiniGen
- IceCache
- PRKV

The most useful comparison question is:

- how does each system define and trigger retrieval under different workload shapes?

LouisKV is especially valuable because its evaluation distinguishes workload geometries instead of collapsing them.

## Placement in the design space

This paper belongs to:

- retrieval-centric KV serving
- long-output reasoning-aware evaluation
- system-overhead-conscious retrieval design

## Project takeaway

LouisKV adds an important benchmarking lesson to the project:

**“long-context KV serving” is not one workload. Long-input and long-output settings stress retrieval systems differently, and the evaluation should preserve that distinction.**
