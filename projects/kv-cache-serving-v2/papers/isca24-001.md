# isca24-001 — Splitwise

## Why it matters in this project

Splitwise is an early architectural predecessor to DistServe-style phase disaggregation. It is important less for KV-specific algorithms and more for the idea that prefill and decode belong to different hardware and operational regimes.

## Architectural role

**Role**: early phase-splitting architecture with state-transfer awareness.

Compared with later papers, Splitwise is still relatively early in KV abstraction. It does not yet build a fully KV-centric serving platform. Instead, it identifies that:

- prompt computation is compute-heavy
- token generation is memory-heavy
- the state between them is worth moving if hardware specialization gains are large enough

Architecturally, this is an early step toward treating KV/state transfer as a systems boundary.

## What architectural problem it solves

The problem is not simply memory footprint. It is phase mismatch:

- the two phases have different hardware demands
- decode wastes high-end compute if kept on the same hardware path as prefill
- colocated designs force compromise

Splitwise solves this by treating phase specialization as the main optimization lever.

## Benchmarking structure

This paper is benchmarked as a **cluster design / hardware allocation** paper, not a KV retrieval or KV reuse paper.

### Scenario

- large-scale LLM inference clusters
- split prefill and decode onto separate machines
- evaluate same-type and mixed-type machine configurations

### Workload shape

- generative inference requests with distinct prefill and decode phases
- emphasis on serving throughput rather than long-context retrieval behavior

### Metrics

- throughput
- cost
- power

### Why this matters for later comparison

Splitwise should be compared to papers that change **phase decomposition** or **resource specialization**, not directly to papers whose main contribution is KV compression or retrieval quality.

It is most comparable to:

- DistServe
- Prefill-as-a-Service

It is less directly comparable to:

- LouisKV
- IceCache
- TRIM-KV

## Placement in the design space

This paper belongs to:

- phase-level disaggregation
- hardware specialization
- state-transfer-aware serving

## Project takeaway

Splitwise helps establish a later comparison rule for the project:

**If a paper’s main architectural move is phase separation, evaluate it by SLO, throughput, cost, and transfer overhead, not by token-retention accuracy alone.**
