# arxiv26-001 — Prefill-as-a-Service

## Why it matters in this project

PrfaaS matters because it pushes phase disaggregation beyond a single tightly coupled cluster. It asks whether prefill can become a remotely provisioned service once KV size drops enough and scheduling becomes selective enough.

## Architectural role

**Role**: cross-cluster / cross-datacenter prefill service architecture.

This is a stronger architectural move than DistServe or Splitwise:

- phase disaggregation is no longer only intra-cluster
- KV transfer is no longer assumed to stay inside one fast network domain
- the system introduces selective offloading rather than universal offloading

The key idea is that KV efficiency at the model layer changes what deployment boundary is feasible at the system layer.

## What architectural problem it solves

The main problem is elasticity under heterogeneous infrastructure:

- prefill and decode capacities scale differently
- workloads are bursty and skewed
- prefix caches are unevenly distributed
- cross-cluster bandwidth is variable

PrfaaS treats long-context prefill as an externalizable service, but only when the system can do so selectively and bandwidth-aware.

## Benchmarking structure

This paper is benchmarked as a **deployment-architecture** paper.

### Scenario

- cross-datacenter or cross-cluster prefill/decode serving
- heterogeneous accelerator deployment
- commodity Ethernet rather than mandatory shared RDMA fabric

### Workload shape

- bursty real serving workloads
- skewed request lengths
- uneven prefix-cache distribution
- long-context requests are especially important

### Metrics

- serving throughput
- P90 TTFT
- cost-normalized throughput
- bandwidth feasibility / modest WAN bandwidth consumption

### Why this matters for later comparison

PrfaaS should mainly be compared to:

- Splitwise
- DistServe
- Mooncake

The comparison question is:

- how far outward can the phase boundary move?
- under what workload and bandwidth assumptions?

It should not be compared directly to retrieval papers purely on raw speedup.

## Placement in the design space

This paper belongs to:

- phase-disaggregated serving
- cross-cluster deployment
- bandwidth-aware KV transfer architecture

## Project takeaway

PrfaaS suggests a useful architectural inflection:

**when KV footprint falls enough, the key systems question becomes not only “can we disaggregate?” but “how far can we push disaggregation before network variability dominates?”**
