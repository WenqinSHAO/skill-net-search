# nsdi26-020 — JITServe

## Why it matters in this project

JITServe matters because it treats SLO diversity and imperfect request information as a first-class serving problem.

## Architectural role

**Role**: SLO-aware scheduler for diverse LLM serving workloads.

It is not a core KV mechanism paper, but it belongs in scope because KV-heavy systems ultimately serve mixed workloads under latency and dependency constraints.

## Benchmarking structure

### Scenario

- diverse application-level SLOs
- incomplete or uncertain request information

### Workload shape

- latency-sensitive streaming chat
- deadline-sensitive requests
- compound or dependent LLM call patterns

### Metrics

- goodput under SLO constraints
- tokens or requests meeting SLOs
- robustness under uncertain request information

## Comparison use

Compare JITServe with:

- DistServe
- DualMap
- Agentix

Its role is to provide the scheduling context in which KV-heavy architectures must operate.

## Project takeaway

JITServe shows that KV-serving architecture must often be evaluated under **imperfect observability of future work**, not only idealized traces.
