# arxiv25-003 — Tokencake

## Why it matters in this project

Tokencake matters because it treats multi-agent serving as both a space-management and time-management problem for KV state.

## Architectural role

**Role**: KV-centric multi-agent serving framework.

Unlike simple prefix reuse systems, Tokencake co-optimizes:

- memory contention among agents
- idle-time reuse during tool-call stalls
- scheduling under agent-aware semantics

## Benchmarking structure

### Scenario

- multi-agent applications with external function calls

### Workload shape

- agent KV caches contend for memory
- long-running tool calls create idle KV residency windows

### Metrics

- end-to-end latency
- effective GPU memory utilization
- representative multi-agent benchmark performance

## Comparison use

Compare Tokencake with:

- KVFlow
- Agentix
- Mooncake

The useful distinction is whether the paper treats KV pressure as:

- workflow prediction
- memory partitioning and offload timing
- program-level scheduling

## Project takeaway

Tokencake pushes the project toward a broader claim:

**in agentic workloads, KV architecture must reason about stalled time as much as occupied space.**
