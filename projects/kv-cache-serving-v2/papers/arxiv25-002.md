# arxiv25-002 — KVFlow

## Why it matters in this project

KVFlow is important because it makes reuse policy workflow-aware rather than request-local.

## Architectural role

**Role**: workflow-aware cache policy for agentic prefix reuse.

Its contribution is not simply “reuse fixed prompts.” It uses expected future workflow steps to decide which KV states should survive eviction.

## Benchmarking structure

### Scenario

- LLM-based multi-agent workflows
- repeated invocation of agents with fixed or semi-fixed prompts

### Workload shape

- structured agent execution schedules
- future reuse is not random; it follows workflow graphs

### Metrics

- cache hit / miss behavior
- recomputation or swapping overhead
- end-to-end workflow efficiency

## Comparison use

Compare KVFlow with:

- LMCache
- Tokencake
- Agentix

It is a reuse-policy paper, but specifically one where **future execution structure** matters.

## Project takeaway

KVFlow shows that for agentic systems, cache policy may need to be **program-aware**, not merely recency-aware.
