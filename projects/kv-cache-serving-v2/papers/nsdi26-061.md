# nsdi26-061 — Agentix

## Why it matters in this project

Agentix matters because it treats programs, not isolated requests, as the scheduling unit for agentic serving.

## Architectural role

**Role**: program-aware serving engine for agentic workloads.

It does not primarily compress or move KV, but it changes the higher-level scheduling model in which agentic KV reuse and waiting behavior occur.

## Benchmarking structure

### Scenario

- LLM agents as general programs
- dependencies among calls inside a larger program

### Workload shape

- program-level head-of-line blocking
- single-threaded and distributed agent programs

### Metrics

- end-to-end program latency
- cumulative wait-time reduction
- benefits of program-aware scheduling

## Comparison use

Compare Agentix with:

- KVFlow
- Tokencake
- JITServe

The key distinction is whether optimization happens at:

- cache policy level
- agent memory/scheduler level
- program scheduling level

## Project takeaway

Agentix shows that once agentic workflows are first-class, some KV-related inefficiencies are better understood as **program scheduling problems** than as local cache problems.
