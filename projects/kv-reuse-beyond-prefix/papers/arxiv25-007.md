# arxiv25-007 — CentroidKV

## Why it matters in this project

CentroidKV is one of the most relevant additions for this project because it uses similarity structure inside KV state itself to reduce what must be retained and reused.

## Reuse-contract view

**Role**: clustering-based approximate retained-state reuse.

Its key move is to merge similar KV cache entries into centroids. That means the system is no longer preserving exact historical state for future computation. It is preserving a clustered approximation that is still useful enough to stand in for the original.

## Beyond-prefix angle

CentroidKV is important because it extends the project beyond exact prefix reuse without jumping directly to semantic prompt matching or cross-model reuse.

The widening happens inside the retained state:

- similar KV entries are treated as approximately interchangeable
- the reused object becomes a centroid rather than an exact prior token state

## Storage-versus-computation trade

CentroidKV helps because it attacks both sides of the trade at once:

- lower storage footprint through clustering
- lower future computation because clustered retained state can still be reused during decoding

The main question it raises is:

- how much fidelity can be lost before approximate centroid reuse stops paying off?

## Comparison use

Compare CentroidKV with:

- `PRKV`
- `SamKV`
- `HySparse`

The useful distinction is whether reuse is widened by:

- restructuring page access
- merging similar retained KV
- sparsifying multiple-context KV
- sharing KV between full and sparse attention layers
